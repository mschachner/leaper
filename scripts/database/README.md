# Leap group database

Generates the static datasets behind Leaper's Database page: every simple
graph on 1–8 vertices (13,598 graphs), with:

- all hops (count), whether a hop exists
- leap-redundant edges (edges used by no hop, as indices into the sorted
  edge list) and whether the graph is leap-connected (still connected after
  removing them)
- Λ₁ and Λ₂ (order, GAP `StructureDescription`, abelianness), reclusiveness
- graph properties: connectivity, bipartiteness, planarity, regularity,
  girth, diameter, triangle count, degree sequence
- names/aliases for recognized graphs (families, small classics, disjoint
  unions of named components)
- precomputed 2D layouts for thumbnails

Sources and IDs: the networkx Graph Atlas for n ≤ 7 (Read–Wilson numbering,
ids `G1`–`G1252`) and nauty's `geng` for n = 8 (ids are canonical graph6
strings). Output is split into two files so the frontend can lazy-load:

- `frontend/src/data/leapDatabaseCore.json` — n ≤ 7 (~0.7 MB), bundled
- `frontend/public/data/leapDatabaseN8.json` — n = 8 (~7.5 MB raw,
  ~0.6 MB gzipped), fetched in the background

Λ₂ is computed as the subgroup generated by `{s·s₀} ∪ {s₀⁻¹·s}` over all
hops `s`, for a fixed hop `s₀` — every even-length word in hops is a
product of such pairs. Λ₁ is reclusive iff |Λ₁| = |Λ₂|.

## Regenerating

Requirements: Python 3.10+, `networkx`, `scipy`, nauty's `geng`
(`brew install nauty`; pass `--geng $(which geng)`), and GAP
(GAP's `StructureDescription` matches Sage's `structure_description()`, so
notation is consistent with the Leaper backend).

```bash
W=/tmp/leapdb

# 1. Enumerate graphs, hops, properties, names, layouts (resumable: rerun
#    until it prints DONE; use --budget SECONDS to bound each run)
python3 build_database.py prepare --workdir $W --geng $(which geng)

# 2. Split GAP input into chunks, then run GAP on each (~4 min total)
python3 build_database.py gapsplit --workdir $W --chunks 40
for k in $(seq 0 39); do gap -q -A /dev/null $W/gaprun_$k.g; done

# 3. Merge into the two frontend datasets
python3 build_database.py merge --workdir $W \
    --out-core ../../frontend/src/data/leapDatabaseCore.json \
    --out-n8   ../../frontend/public/data/leapDatabaseN8.json

# 4. Verify against known results from the paper
python3 verify_database.py \
    ../../frontend/src/data/leapDatabaseCore.json \
    ../../frontend/public/data/leapDatabaseN8.json
```

`verify_database.py` checks the dataset against the paper: Hall's condition
vs. hop existence, Cor 2.4 (independence bound), [Λ₁:Λ₂] ≤ 2, the Kₙ and Cₙ
computations (including K8 → S8 with D(8) = 14,833 hops and
Λ₁(C8) ≅ Z4≀Z2), Prop 3.15/3.16 (abelian orders, odd-order
classification), Theorem 3.14 (disjoint-union orders, checked on all 1,216
two-component graphs), and the leap-redundancy invariants.

## Group display names

The frontend (`frontend/src/lib/leapDatabase.js`, `GROUP_DISPLAY`) renders
friendlier names for GAP's clunkier `StructureDescription` strings — e.g.
`(C2 x C2 x C2 x C2) : ((S3 x S3) : C2)` displays as `S4 ≀ C2`. Since
`StructureDescription` is not injective (non-isomorphic groups can share a
string), each identification was verified with GAP `IdGroup` against every
occurrence (or a sample plus a structural argument, for the order-1152
class) in the database. If the corpus changes, re-verify before trusting
the aliases.

## Beyond n = 8

n = 9 has 274,668 graphs (~40× the data) and K9 has 133,496 hops; static
JSON stops being reasonable there. A server-backed database (SQLite +
a search endpoint) would be the next architecture.
